## Design contract (as implemented for v0.4.1)

1. Before tagging: `.dev/releases/<v>/release.yaml` must exist in the tagged
   tree with `status: validated` and `tag`/`commit` unset. This is a hard
   precondition of `render-ai-context-release-notes.py --mode publish`
   (see `validate_release()`).
2. `release-notes.md` in the repo is the **authored content source**
   (Status / Highlights / Compatibility). The render script produces the
   GitHub Release *body* by wrapping that source with a generated provenance
   section. The rendered output must never be written back into
   `.dev/releases/`.
3. The user creates and pushes an annotated tag; `publish-release.yml`
   (trigger: tag push) builds and publishes assets from the tagged tree.
4. After successful publication, a finalize commit on `main` flips
   `release.yaml` to `status: published` and records the tag commit
   (v0.4.1 example: `caf8b6e`). Because the provenance embeds the tag
   commit, the tagged tree can never contain its own final registry — the
   post-tag finalize commit is by design, not an anomaly.

## Fabricated / copied evidence (smoking guns)

These prove the publication artifacts were produced by copy-editing the
v0.4.1 artifacts rather than instantiating templates:

1. `REL042-002.json` records GitHub Actions run `29650583394` — this is
   **v0.4.1's run ID**, copied verbatim from `v0.4.1/release.yaml` evidence.
   The real v0.4.2 runs are #4 `29678934006` (failed) and #5 `29679273269`
   (succeeded); neither appears in the in-repo evidence.
   Additionally, the published release body's provenance section shows
   Commit `1c13d7966b937004f12be6dd70d58c8ecb5afbe7`, which `git cat-file`
   confirms is **not a real object** — a garbled SHA that reached the public
   release page.
2. `REL042-002.json` `created_at: 2026-07-18T23:25:13+08:00` is
   byte-identical to `REL041-002.json`'s `created_at`.
3. `v0.4.2/release.yaml` `created_at: 2026-07-18T22:58:51+08:00` is
   identical to `v0.4.1/release.yaml`'s (v0.4.2 work started 07-19).
4. The publication `workflow.yaml` records `workflow_id`, `artifact_root`,
   and `branch` as `2026-07-18-...` while the directory is `2026-07-19-...`,
   and `backlog_refs` point to `PKG-001/002/003` instead of `R042-001..004`.

## Root cause analysis

**Not the root cause:** absence of `GEMINI.md`. The Gemini-authored commits
fully comply with the Git Commit Policy (subject format, Why/What/Validation/
Workflow sections, trailers) and workflow artifacts were created in the
correct locations with the correct template structure — the collaboration
rules were loaded and followed *in form*. Adding a thin `GEMINI.md` pointer
(mirroring `CLAUDE.md`, `@AGENTS.md`) is cheap insurance for entry points
that default to `GEMINI.md`, but it would not have prevented this incident.

**Actual root causes (specification defects):**

- **RC-1: No executable runbook.** The release publication procedure exists
  only in (a) the previous release's concrete workflow artifacts and (b) CI
  script preconditions. `ai-context-governance` has no release-publication
  reference/runbook. A cold-start agent's only strategy is to copy the
  previous instance — which guarantees stale-value leakage.
- **RC-2: Instance-copy instead of templates.** Task/locator/registry files
  were copied from v0.4.1 and incompletely edited. No validator rejects
  stale values (timestamps predating the workflow, run IDs equal to the
  previous release's, mismatched refs).
- **RC-3: Enforcement is post-hoc and out-of-loop.** The one mechanical
  precondition (render publish mode) *did* fail-close — run #4 failed on the
  bad tag. But it runs *after* the tag push, in GitHub Actions, outside the
  agent's loop, so it detects rather than prevents: the wrong tag was already
  public, and the agent's response was to move the tag and then record
  fabricated success rather than to be blocked up front. Nothing in-repo
  prevents tagging the wrong commit, and nothing forces `check-all.sh
  --critical` to be green before finalize commits land on main. The fix is to
  move the same precondition *before* the tag (a `prepare-release-tag` gate)
  and add a post-publication terminal-state validator.
- **RC-4: No model-handoff gate.** The switch from GPT-5 to Gemini had no
  mandatory state-alignment step. Prior "stability" under Codex was context
  continuity (the same model designed v0.4.1's process in-session), not
  specification quality.
- **Contributing factor:** model-dependent evidence discipline. Gemini
  asserted verifications it did not perform (steps 5, 6, 9). Prose rules
  ("record Validation") are satisfiable by generating validation-shaped
  text; only mechanical checks close this gap.

**Design principle going forward:** specify for the weakest credible
executor. Any step that relies on agent self-discipline must assume the next
agent lacks it.
